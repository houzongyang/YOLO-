# 智慧停车系统修改后论文写作说明文档

## 一、本次修改背景

本次项目修改的直接背景是：导师指出原系统存在“基础功能不够完善、车位状态变化过程不清楚、创新点不足”等问题。原系统虽然已经具备车牌识别、停车记录、车位区域管理等基础模块，但在论文和答辩表达中容易被理解为普通后台管理系统，智慧停车特征不够突出。

因此，本次修改围绕“识别结果如何驱动车位业务变化”这一主线进行完善，重点解决以下问题：

1. **车位状态变化不清楚**：补充并强化车辆入场、出场时空余车位和已用车位的联动逻辑。
2. **创新表达不足**：新增智慧停车相关能力，包括车位智能推荐、异常预警、高峰预测、动态计费、可解释数据大屏、智能问答等。
3. **页面展示不完整**：优化车位管理页和数据大屏，使空余车位、已用车位、占用率、推荐区域、预警信息等能够直观展示。
4. **系统流程不闭环**：将 YOLO 车牌识别结果与停车记录、车位状态、计费结算、数据统计联动起来，形成“识别—记录—联动—展示—辅助决策”的完整流程。
5. **演示数据不严谨**：清理和避免依赖伪造车辆数据，强调系统展示应基于数据库中的真实停车记录和真实车位区域数据。

本次修改后的系统不应再只描述为“车牌识别管理系统”，更适合在论文中表述为“基于 YOLO 车牌识别与业务联动的智慧停车管理系统”。

## 二、本次新增和强化的功能

### 2.1 车位状态联动功能

**实现状态：已实现。**

系统在车位区域中维护 `used`、`unused`、`areaStatus`、`price` 等字段。其中：

- `used` 表示已用车位数量；
- `unused` 表示空余车位数量；
- `areaStatus` 表示区域状态，空余车位为 0 时为已满，否则为可用；
- `price` 表示该区域基础停车单价。

车辆入场时，系统根据车牌号判断该车辆没有未结束停车记录，则创建新的停车记录，并将推荐停车区域的空余车位减 1、已用车位加 1。车辆出场时，系统根据同一车牌号找到未结束停车记录，补充出场时间、计算停车时长和费用，同时释放车位，即空余车位加 1、已用车位减 1。

该功能解决了“空余车位和已用车位如何变化”的问题，使车位管理不再只是静态录入，而是随车辆进出动态变化。

### 2.2 YOLO 识别业务联动

**实现状态：已实现。**

系统使用 YOLO 模型对图像、视频或摄像头画面中的车牌区域进行检测，再通过 LPRNet 对车牌字符进行识别。识别成功后，系统将车牌号提交给后端停车记录接口，由后端完成入场或出场判断。

需要注意的是，YOLO 主要负责车牌目标检测，LPRNet 负责车牌字符识别，车辆调度、车位推荐、停车计费等属于后端业务逻辑。论文中应明确区分“视觉识别层”和“停车业务层”，避免把所有功能都归因于 YOLO 模型本身。

本次修改强化了识别结果与停车业务之间的关系：识别到车牌后，不仅展示检测结果，还能进一步驱动停车记录和车位状态变化，从而形成完整业务闭环。

### 2.3 车位智能推荐

**实现状态：已实现基础版，可继续扩展。**

系统新增车位推荐能力，根据区域空余车位、占用率、停车单价、近期周转情况等因素计算推荐分，为车辆推荐更适合停入的区域。

推荐逻辑主要考虑：

- 空余车位越多，推荐优先级越高；
- 占用率越低，等待风险越低；
- 单价越低，停车成本越低；
- 近期周转情况可反映区域使用活跃度。

前端在数据大屏、图像识别结果页、摄像头识别页中展示推荐区域、空余车位、占用率、等待风险、预估费用等信息。后端在车辆入场时也可以优先使用推荐结果选择停车区域。

论文中可将其写作“基于实时车位状态和停车成本的轻量级车位推荐策略”。

### 2.4 异常停车与高占用预警

**实现状态：已实现计算型预警基础，可扩展为持久化预警。**

系统新增预警能力，用于发现停车场运行中的异常或高风险状态，包括：

- 区域占用率过高，提示高占用风险；
- 车辆长时间未出场，提示长时停车；
- 当前没有异常时，页面显示暂无真实预警，不再使用伪造数据展示。

当前预警结果主要由后端根据车位区域和停车记录实时计算得到，适合用于数据大屏展示。后续可增加预警表，将预警信息保存到数据库，并支持管理员处理、标记、追踪。

### 2.5 停车高峰预测

**实现状态：已实现轻量预测，不属于深度学习预测模型。**

系统新增未来停车压力预测能力，根据历史停车记录中的入场、出场时间，估算未来若干小时的入场数、出场数、预计空余车位和压力等级。

该功能用于辅助管理员提前判断停车压力。例如，当预测未来一段时间空余车位持续下降时，系统可以提示管理员加强入口引导或开放备用区域。

论文中应准确表述为“基于历史停车记录的规则统计与趋势预测”，不要写成深度学习预测模型或复杂时序神经网络模型。

### 2.6 动态计费

**实现状态：已实现基础动态计费规则，可继续扩展。**

系统原有停车计费主要依据停车时长和区域单价。本次修改后，计费逻辑进一步考虑免费时长、高峰系数、占用率系数和封顶价等因素。

当前计费思路如下：

- 设置一定免费停车时长；
- 根据停车区域单价和计费时长计算基础费用；
- 在早晚高峰时段使用高峰系数；
- 根据区域占用率调整费用系数；
- 设置单日封顶价，避免费用异常过高。

出场结算时，后端会根据区域信息、停车时长和当前时间计算最终费用。论文中可将其作为“面向车位周转率的动态计费策略”。

### 2.7 可解释数据大屏

**实现状态：已实现。**

数据大屏不再只展示简单图表，而是增加了智慧停车运行状态展示，包括：

- 总车位、空余车位、占用车位、当前在场车辆；
- 今日收入、今日出场次数、平均停车时长、周转率；
- 车位智能推荐结果；
- 未来停车压力预测；
- 异常停车和高占用预警；
- 区域使用情况和推荐分；
- 系统根据当前状态生成的解释性文字。

这些内容能够帮助管理员快速理解当前停车场状态，而不是只看到原始数据。论文中可表述为“面向停车运营管理的可解释数据看板”。

### 2.8 智能问答增强

**实现状态：已实现业务问答基础，可继续扩展。**

系统新增 `/smart/chat` 接口，用于回答与停车业务相关的问题，例如：

- 当前还有多少空余车位；
- 今日收入情况；
- 推荐停入哪个区域；
- 当前是否存在预警。

当前智能问答更适合描述为“基于关键词和业务接口的停车运营问答”，而不是完整的大模型智能体。后续可以扩展为接入大模型，并通过工具调用方式查询停车业务接口。

## 三、本次优化的问题

### 3.1 页面滚动与展示问题

针对数据大屏、车位管理等页面内容增多后无法完整查看的问题，对页面布局和滚动区域进行了优化，使大屏卡片、图表、表格、预警和推荐信息能够正常浏览。

论文中可简单描述为：系统优化了前端页面布局，使智慧停车数据展示更加完整、清晰，提升了后台管理端的可用性。

### 3.2 JSON 递归问题

后端接口返回复杂对象时，容易出现 JSON 序列化递归或返回结构不清晰的问题。本次修改将智慧停车相关数据通过 `Map`、列表和基础字段进行组装返回，避免前端接收复杂实体关系造成解析异常。

论文中不需要展开具体报错细节，只需说明：系统对后端接口返回结构进行了规范化处理，保证前后端数据交互稳定。

### 3.3 假数据和演示数据清理

本次修改强调不再依赖伪造车辆数据、模拟车牌或固定演示记录。数据大屏、预警、推荐和问答应基于数据库中的真实车位区域、真实停车记录和真实识别结果生成。

当真实历史记录不足时，系统应显示“暂无真实历史停车记录，数据不足，暂不预测”等提示，而不是自动填充虚假数据。这一点在论文中可以体现为系统数据来源更加严谨。

### 3.4 车位统计口径优化

系统新增或强化了车位统计能力，能够统计总车位、空余车位、已用车位、满载区域、占用率等指标，并在车位管理页和数据大屏中展示。

这样可以直接回答导师提出的“空余车位、占用车位如何变化”的问题：系统通过车辆入场和出场事件自动修改车位数量，并通过统计接口实时汇总展示。

### 3.5 页面展示优化

前端页面新增了推荐卡片、预警卡片、预测图表、关键指标卡片、占用率进度条、推荐分等内容，使系统创新点能够在答辩演示中被直观看到。

## 四、新增后端模块与接口

### 4.1 SmartParkingController

`SmartParkingController` 是本次新增的智慧停车统一接口控制器，路径前缀为 `/smart`。它将推荐、预测、预警、动态计费、智能问答等能力统一暴露给前端。

主要接口包括：

| 接口 | 方法 | 功能 |
| --- | --- | --- |
| `/smart/summary` | GET | 获取智慧停车综合摘要，包括车位、收入、推荐、预测、预警和解释文本 |
| `/smart/recommend` | GET | 根据当前车位状态计算推荐停车区域 |
| `/smart/forecast` | GET | 获取未来停车压力预测结果 |
| `/smart/alerts` | GET | 获取异常停车和高占用预警 |
| `/smart/alerts/{id}/resolve` | POST | 预警处理接口，目前主要用于说明计算型预警刷新机制 |
| `/smart/billing/estimate` | POST | 估算动态停车费用 |
| `/smart/chat` | POST | 停车业务智能问答 |

### 4.2 SmartParkingService

`SmartParkingService` 是本次新增的智慧停车业务服务类，负责具体业务计算，包括：

- 汇总车位区域、在场车辆、今日收入、平均停车时长等运营指标；
- 根据空余率、价格、占用风险、近期周转情况计算推荐区域；
- 根据历史停车记录生成未来若干小时的压力预测；
- 根据占用率和未出场记录生成预警；
- 根据免费时长、高峰系数、占用率系数、封顶价计算费用；
- 根据用户问题返回业务问答结果；
- 生成数据大屏中的解释性文本。

### 4.3 与原有模块的联动

本次修改不是孤立新增模块，而是与原有模块形成联动：

- `PlateRecordsController` 在车辆入场时调用推荐服务选择停车区域；
- `PlateRecordsController` 在车辆出场时调用动态计费服务计算费用；
- `AreaController` 提供车位统计和区域状态维护能力；
- Flask 识别服务将车牌号提交给后端停车记录接口，触发入场或出场逻辑；
- 前端数据大屏和识别页面调用 `/smart` 接口展示智慧停车结果。

## 六、修改后的系统整体流程

### 6.1 入场流程

1. 车辆到达入口，系统通过图片、视频或摄像头获取车辆画面。
2. YOLO 模型检测车牌区域，LPRNet 识别车牌字符。
3. 识别成功后，系统将车牌号提交给后端停车记录接口。
4. 后端查询该车牌是否存在未结束停车记录。
5. 如果不存在未结束记录，则认为车辆入场。
6. 后端调用智能推荐服务，选择可用停车区域。
7. 系统新增停车记录，设置入场时间，出场时间为空。
8. 推荐区域空余车位减 1，已用车位加 1，区域状态自动更新。
9. 前端车位管理页和数据大屏刷新后展示最新状态。

### 6.2 出场流程

1. 车辆到达出口后再次被识别到车牌号。
2. 后端根据车牌号查询是否存在未结束停车记录。
3. 如果存在未结束记录，则认为车辆出场。
4. 系统写入出场时间，计算停车时长。
5. 动态计费模块根据区域单价、停车时长、高峰系数、占用率系数和封顶价计算费用。
6. 系统更新停车记录中的出场时间、停车时长和费用。
7. 对应区域空余车位加 1，已用车位减 1，区域状态自动更新。
8. 数据大屏和停车记录页面展示最新出场记录和费用。

### 6.3 推荐流程

1. 系统读取所有车位区域状态。
2. 过滤无效区域或已满区域。
3. 根据空余率、价格、低占用风险、近期周转情况计算推荐分。
4. 返回推荐区域、推荐原因、空余车位、占用率、等待风险、预估费用。
5. 前端在大屏、图像识别页或摄像头识别页展示推荐结果。

### 6.4 预警流程

1. 系统读取车位区域和停车记录。
2. 判断是否存在高占用区域。
3. 判断是否存在长时间未出场车辆。
4. 生成实时预警结果。
5. 前端数据大屏展示预警信息；如果没有真实预警，则显示暂无真实预警。

### 6.5 计费流程

1. 出场时计算入场时间与出场时间之间的停车时长。
2. 根据免费时长得到计费时长。
3. 根据区域单价计算基础费用。
4. 结合高峰时段和区域占用率进行系数调整。
5. 应用单日封顶价。
6. 得到最终停车费用并写入停车记录。

### 6.6 数据大屏流程

1. 前端调用 `/smart/summary`。
2. 后端汇总车位、停车记录、收入、推荐、预测、预警等信息。
3. 前端展示关键指标、推荐区域、预测图表、预警列表和解释文本。
4. 管理员通过大屏了解停车场运行状态。

## 七、论文可写创新点

### 7.1 识别结果驱动的停车业务闭环

系统将 YOLO 车牌检测、LPRNet 车牌识别与停车业务逻辑结合。识别结果不只是展示在页面上，而是进一步触发停车记录创建、出场结算、车位占用和释放。这是本系统区别于普通车牌识别演示系统的重要创新点。

### 7.2 基于实时车位状态的智能推荐

系统根据区域空余率、占用率、停车单价和近期周转情况计算推荐区域，为车辆提供停车引导建议。该方法实现成本较低，适合本科毕业论文，同时能体现智慧停车的调度思想。

### 7.3 车位状态动态联动机制

系统围绕车辆入场和出场事件自动维护空余车位、已用车位和区域状态，解决传统停车管理中人工维护车位状态不及时的问题。

### 7.4 异常停车与高占用预警

系统根据真实停车记录和区域占用率生成预警信息，帮助管理员及时发现高占用区域和长时间未出场车辆，提升停车场管理效率。

### 7.5 轻量级停车压力预测

系统利用历史停车记录进行未来停车压力估计，帮助管理员提前判断未来停车供需变化。该功能虽然不是复杂深度学习模型，但能够体现数据驱动的管理思想。

### 7.6 可解释运营数据大屏

系统不仅展示图表，还生成解释性文字，帮助管理员理解当前占用率、推荐区域和预警状态，使数据展示更接近运营决策支持。

### 7.7 面向业务的智能问答

系统通过停车业务接口回答管理员问题，将车位统计、收入统计、推荐和预警能力以问答方式呈现，提升系统交互体验。

## 八、论文中不要夸大的点

为保证论文表述严谨，以下内容不建议夸大：

1. **不要写“改进了 YOLOv11 网络结构”**。本项目没有修改 YOLOv11 的 Backbone、Neck 或 Head，更准确的表述是“基于 YOLO 模型进行车牌检测，并将检测结果接入停车业务流程”。
2. **不要写“提出了新的深度学习预测模型”**。停车高峰预测属于基于历史记录的统计和规则预测，不是 LSTM、Transformer 等深度学习预测模型。
3. **不要写“实现了完整自动驾驶式车辆调度”**。系统实现的是停车区域推荐和车位状态联动，不涉及车辆路径规划或自动驾驶控制。
4. **不要写“预警数据来自模拟数据”**。论文中应强调预警来自真实车位状态和停车记录，真实数据不足时应显示暂无数据。
5. **不要写“智能问答已经具备完整大模型推理能力”**。当前更适合写作业务问答助手，后续可扩展为大模型工具调用。
6. **不要写“进口出口已经具备复杂多摄像头轨迹追踪”**。当前主要通过车牌号和未结束停车记录判断入场、出场；多摄像头方向、点位和防重复识别可作为扩展。

## 九、答辩话术

### 9.1 回答“系统创新在哪里”

本系统的创新点不是单独做一个车牌识别页面，而是把车牌识别结果和停车业务管理联动起来。YOLO 负责检测车牌区域，LPRNet 负责识别车牌号码，后端根据车牌号判断车辆是入场还是出场，并自动更新停车记录、空余车位、已用车位和停车费用。同时系统增加了车位智能推荐、异常预警、停车压力预测、动态计费和可解释数据大屏，使系统从普通管理后台升级为具备辅助决策能力的智慧停车系统。

### 9.2 回答“空余车位和已用车位如何变化”

系统中每个停车区域都有空余车位和已用车位。车辆入场时，如果该车牌没有未结束停车记录，系统会创建入场记录，并将推荐区域的空余车位减 1、已用车位加 1；车辆出场时，如果该车牌存在未结束记录，系统会写入出场时间、计算费用，并将对应区域空余车位加 1、已用车位减 1。区域状态也会根据空余车位是否为 0 自动显示为可用或已满。

### 9.3 回答“YOLO 是检测车牌还是调度车辆”

YOLO 主要负责检测车牌区域，不直接做车辆调度。系统中的车辆调度或推荐是后端根据车位空余率、占用率、价格和周转情况计算出来的。也就是说，YOLO 是感知层，后端智慧停车模块是业务决策层。

### 9.4 回答“对 YOLO 做了什么创新”

本项目没有修改 YOLO 的网络结构，而是将 YOLO 车牌检测与 LPRNet 字符识别结合，并进一步接入停车管理业务，实现从车牌识别到停车记录、车位联动、计费结算和数据展示的完整流程。创新点主要体现在场景化集成和业务闭环，而不是底层模型结构创新。

### 9.5 回答“一个进口一个出口能否联动”

可以联动。当前系统是根据车牌号和未结束停车记录判断入场或出场：没有未结束记录时视为入场，有未结束记录时视为出场。只要入口和出口识别设备调用同一个后端接口、使用同一张停车记录表，就能实现联动。后续可以增加摄像头类型字段，明确区分入口摄像头和出口摄像头，使逻辑更加严谨。

## 十、可直接放进论文的章节写法

### 10.1 系统功能完善章节示例

本系统在原有车牌识别和停车记录管理基础上，进一步完善了智慧停车业务闭环。系统通过 YOLO 模型检测车牌区域，并结合 LPRNet 完成车牌字符识别。识别结果传入后端停车管理模块后，系统根据车牌号判断车辆入场或出场，并自动维护停车记录、车位占用状态和停车费用。相比传统人工录入方式，该流程减少了人工操作，提高了停车场出入口管理效率。

在车位管理方面，系统为每个停车区域维护空余车位、已用车位、区域状态和停车单价。车辆入场时，系统自动减少对应区域的空余车位并增加已用车位；车辆出场时，系统自动释放车位并完成费用结算。区域状态根据空余车位数量动态更新，从而保证车位状态与实际停车记录保持一致。

### 10.2 智慧停车创新章节示例

为提高系统的智慧化程度，本文设计并实现了智慧停车辅助决策模块。该模块主要包括车位智能推荐、异常停车预警、停车压力预测、动态计费、可解释数据大屏和业务智能问答等功能。车位智能推荐根据区域空余率、占用率、停车价格和近期周转情况计算推荐分，为车辆选择更合适的停车区域。异常预警模块根据高占用区域和长时间未出场记录生成提示信息，辅助管理员及时发现运行风险。停车压力预测模块基于历史停车记录估计未来短时间内的车位压力，为停车场调度提供参考。

数据大屏模块对车位数量、空余车位、占用车位、今日收入、平均停车时长、推荐区域和预警信息进行集中展示，并生成解释性文字，帮助管理员快速了解停车场运行状态。智能问答模块通过业务接口回答停车场运营相关问题，提升系统交互体验。

### 10.3 技术路线章节示例

系统整体采用前后端分离架构。前端基于 Vue 和 Element Plus 实现后台管理页面，使用 ECharts 展示停车场运行数据；后端基于 Spring Boot 和 MyBatis-Plus 实现停车记录、车位区域和智慧停车接口；算法服务基于 Flask 部署 YOLO 车牌检测和 LPRNet 字符识别。前端通过 HTTP 接口调用后端服务，后端通过接口调用算法服务，并将识别结果与停车业务数据进行联动。

系统的数据流为：车辆图像首先进入算法服务进行车牌检测与识别，识别结果传入后端停车管理模块；后端根据车牌号判断入场或出场，并更新停车记录和车位区域状态；智慧停车模块进一步基于车位和停车记录数据生成推荐、预测、预警和统计结果；前端页面将这些结果以表格、图表、卡片和问答形式展示给管理员。

### 10.4 不足与展望章节示例

当前系统已经实现了车牌识别与停车业务联动、车位状态动态更新、车位推荐、预警、预测和数据大屏展示等功能，但仍有进一步优化空间。首先，YOLO 模型主要作为车牌检测模型使用，尚未对网络结构进行改进，后续可针对停车场场景采集更多数据进行模型微调。其次，停车压力预测目前采用轻量级统计方法，后续可引入时间序列模型提高预测准确性。再次，入口和出口联动当前主要依赖车牌号和停车记录状态，后续可增加摄像头点位和方向字段，完善多入口多出口场景下的防重复识别机制。最后，智能问答模块可进一步结合大模型和工具调用，实现更自然的停车运营分析能力。

## 十一、论文修改建议

建议用户在毕业论文中重点修改以下章节：

1. **需求分析章节**：增加导师关注的车位状态联动、停车推荐、预警、预测和运营看板需求。
